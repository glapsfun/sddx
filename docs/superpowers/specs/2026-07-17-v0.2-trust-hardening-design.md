# sddx v0.2 — Trust Hardening (design)

**Date:** 2026-07-17
**Status:** approved design, pre-implementation
**Theme:** harden the two identity claims of sddx — provable completion and
hard-block TDD — and extend receipt trust beyond the local machine.

Seven features in three bundles. All are deterministic-gate work: no new
model judgment anywhere, only new evidence and new checks over it.

---

## A. Prove the oracle

The framework's guarantee rests on the oracle discriminating between done and
not-done. Today an oracle that always passes sails through. This bundle makes
the oracle itself provable.

### A1. Oracle red-check

New CLI step: `sddx red-check <task-id>`.

- Runs the task's `oracle.run` once during RED and **requires a non-zero
  exit**. Exit 0 means the oracle cannot fail → it proves nothing.
- Records the result in the task file as `oracle_red`:
  `{ exit_code, timestamp, stdout_sha256, stderr_sha256 }`.
- `verify` hard-rejects a pass when `oracle_red` is missing **or** was
  recorded after the first green test run (timestamps compared against the
  recorder's red→green record). Evidence ordering, not model claims, proves
  the oracle could fail.
- On a pre-passing oracle, the CLI prints: the oracle does not discriminate —
  fix the spec. The task cannot leave RED until a failing oracle run is
  recorded.
- Skill changes: `quick` and `run` loops gain one line — run red-check after
  the failing test exists, before implementation starts.

### A2. Flakiness runs

- Spec gains optional `oracle.runs: N` (integer ≥ 1, default 1).
  userConfig `oracle_runs_default` sets the repo default.
- `verify` executes the oracle N times sequentially. **Pass requires N
  zero-exits.** Any non-zero run fails the verification as a whole.
- Receipt schema bumps to **v3**: the single run record becomes a `runs: []`
  array with `{ exit_code, duration_ms, stdout_sha256, stderr_sha256 }` per
  run. `audit` accepts v2 and v3 so existing chains stay valid.

### A3. Env capture

Receipt v3 additionally records:

```json
"env": {
  "os": "darwin",
  "arch": "arm64",
  "runtime": "bun",
  "runtime_version": "1.2.x",
  "dirty_tree": false
}
```

`dirty_tree` comes from `git status --porcelain` at verify time. Pure
additive evidence; no new judgment.

## B. Close the gate holes

The TDD gate hard-blocks Edit/Write in RED, but Bash is an open bypass
(`sed -i`, `tee`, redirection, heredocs). This bundle closes it and stops
unproductive spinning.

### B1. RED Bash allow-list

- `PreToolUse` hook extends to the Bash tool. In RED phase a command is
  allowed only if the **first token of every pipeline segment** matches the
  allow-list; otherwise it is hard-blocked with an instructive message in the
  same style as the Edit gate.
- Baseline allow-list:
  - test runners: `bun`, `npm`, `npx`, `pnpm`, `yarn`, `pytest`, `go`,
    `cargo`, `make`
  - read-only tools: `ls`, `cat`, `grep`, `rg`, `find`, `head`, `tail`, `wc`
  - `git`, allowed only when its subcommand (first non-flag argument) is one
    of: `status`, `diff`, `log`, `show`
- **Any redirection to a file (`>`, `>>`) is blocked regardless of the
  command.** Deterministic, no target parsing.
- userConfig `red_bash_allow` extends the list (never replaces it).
- The existing per-file `allow: <path>` escape hatch applies and is visible
  in the receipt, unchanged.

### B2. Stuck-loop escalation

- The PostToolUse recorder additionally stores a **failure fingerprint**:
  sha256 over the test exit code plus the normalized last 20 lines of
  combined output (whitespace-collapsed, timestamps stripped).
- Three consecutive identical fingerprints (userConfig `stuck_threshold`,
  default 3) flip the task phase to `stuck`.
- The Stop gate refuses further iteration on a `stuck` task and instructs
  escalation to the human. A distinct fingerprint resets the counter —
  progress is allowed, spinning is not.
- `BOARD.md` shows `stuck` as its own state.

## C. Trust beyond the laptop

Receipts prove things locally; nothing enforces them where merges happen.

### C1. CI receipt gate

- New mode: `sddx audit --ci`. Exit non-zero **only on tamper evidence**:
  - broken receipt hash chain,
  - edited, deleted, or schema-invalid receipts,
  - a task marked `done` without a receipt.
- A repo or PR with no sddx activity passes clean — sddx stays opt-in per
  task; safe to add to any repo.
- Ships with a documented composite GitHub Action example that checks out
  full history and runs the bundled `dist/audit.mjs` directly — zero-install
  (G6), no marketplace dependency.

### C2. Receipt signing

- When `git config user.signingkey` is set and SSH signing is configured,
  `verify` signs the receipt's content hash with `ssh-keygen -Y sign` and
  embeds optional v3 fields `signature` and `signer`.
- `audit` verifies signatures when present (against
  `gpg.ssh.allowedSignersFile` when available) and reports `unsigned`
  otherwise. **Absence never fails an audit** — consistent with C1's
  tamper-only stance. A present-but-invalid signature does fail.

---

## Cross-cutting

- **Receipt v3** is one schema change shared by A2, A3, and C2; it lands as a
  single task first. `audit` reads v2 and v3; `verify` writes only v3.
- **Config additions** (manifest `userConfig`): `oracle_runs_default`,
  `red_bash_allow`, `stuck_threshold`.
- **Hook latency budget** unchanged: the Bash allow-list check is pure string
  work; red-check and flaky runs execute in CLI steps, not hooks.
- Everything else is independent — seven features, roughly seven sddx-sized
  tasks, each with its own oracle.

## Out of scope (unchanged non-goals)

- Strict CI coverage mode (fail PRs whose source changes lack receipts) —
  deferred until tamper-only proves out.
- Cross-machine identity federation, web UI, other harnesses.
